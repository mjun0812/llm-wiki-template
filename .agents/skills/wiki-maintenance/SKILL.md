# wiki-maintenance

`wiki/` 以下の既存ページを読み、知識ベースとして使いやすい形に保守する。
`sources/` から新規ページを作る取り込み作業は範囲外とし、既存Wikiの整理、推敲、統合、移動に集中する。

## 併用するSkill

`wiki/` の本文を作成・更新する前に、local skill の `japanese-tech-writing` と `stop-ai-slop-jp` を読む。
最終本文は、日本語の技術文書として読みやすく、AI臭の少ない文章に整える。

## ルール

- **sources編集禁止**：`sources/` 配下は参照確認だけに使い、frontmatter、本文、画像パス、ファイル名を変更しない。
- **既存情報を失わない**：統合、削除、リネーム、移動を行う前に、差分で失われる見出し、リンク、出典、未確認事項がないか確認する。
- **根拠を残す**：ページのfrontmatter `sources` や本文の「参照元」を維持し、出典のない断定を追加しない。
- **構成変更は最小限**：カテゴリやディレクトリは、検索性や重複解消に明確に効く場合だけ変える。
- **リンクを追随する**：ページをリネーム・移動したら、`wiki/` 内のリンク、frontmatter `related`、`wiki/index.md` を更新する。
- **index/changelog更新**：Wiki本文、配置、索引を実質的に変えたら、`wiki/index.md` と `wiki/changelog.md` も更新する。

## 対象の決め方

ユーザーが対象ページ、カテゴリ、ディレクトリを指定した場合は、その範囲だけ処理する。
指定がない場合は、次を確認して整理対象を決める。

- `wiki/index.md` に載っていないページ
- 同じ主題を扱う重複・近接ページ
- frontmatterの `category`、`related`、`sources` と配置がずれているページ
- 古い名前、曖昧な名前、深すぎる階層にあるページ
- 本文が箇条書きの集積だけになっているページ
- カテゴリ全体を1ページに抱えた巨大なカタログページ (主題単位への分割対象)
- 本文が原資料へのリンク紹介文の羅列になっているページ
- 件数など、原資料の増減で古くなる統計値を本文に持つページ

大規模な移動や統合になりそうな場合は、一度に全体を変えず、カテゴリ単位で区切る。

## 手順

### 1. 現状を読む

- 対象のWikiページ、`wiki/index.md`、`wiki/changelog.md` を読む。
- 必要な場合だけ、frontmatter `sources` や本文の「参照元」から原資料を読む。
- 重複、リンク切れ、索引漏れ、カテゴリずれ、本文品質の問題を整理する。

### 2. 方針を決める

対象ごとに次のどれを行うか決める。

- **推敲**：本文の見通し、見出し、段落、表現を直す。情報の意味は変えない。
- **統合**：重複ページを1ページにまとめる。統合先に出典と関連リンクを引き継ぐ。
- **分割**：1ページが複数主題を抱えている場合だけ分ける。
- **移動・リネーム**：カテゴリ構成や名前が検索性を下げている場合だけ行う。
- **索引修正**：`wiki/index.md` のカテゴリ、リンク、説明を実態に合わせる。

迷う場合は、削除や大幅移動よりも本文改善とリンク整理を優先する。

### 3. Wikiページを更新する

既存ページを編集する場合は、frontmatterを保つ。
新規ページが必要な場合は `_template/wiki.md` を使う。

frontmatterは次を確認する。

- `title` はページ内容を端的に表す。
- `category` は `wiki/index.md` の主要カテゴリに合わせる。
- `sources` は統合元ページや原資料から失わずに引き継ぐ。
- `related` は実際に参照価値がある既存Wikiページだけにする。
- `updated` は実質更新日に合わせる。

本文は次の方針で直す。

- 結論、背景、具体、未確認事項を分ける。
- 箇条書きだけで意味が取りにくい箇所は、短い本文に直す。
- 出典のない一般化や断定を避ける。
- 原資料同士や既存ページ同士が矛盾する場合は、片方に寄せず違いを明示する。

### 4. 移動・リネーム時の処理

ページを移動・リネームする場合は、可能なら `git mv` を使う。
その後、次を確認して更新する。

- `wiki/**/*.md` の本文リンク
- frontmatter `related`
- `wiki/index.md`
- 統合元ページに残す必要がある情報

統合でページを削除する場合は、統合先に旧ページの要点、出典、関連リンクが残っていることを確認してから削除する。

### 5. indexを更新する

`wiki/index.md` を実態に合わせて更新する。
カテゴリ説明を大きく書き換えない。
カテゴリを新設、統合、削除する場合は、最後の報告に理由を書く。

### 6. changelogを更新する

`wiki/changelog.md` の先頭に履歴を追記する。
形式は既存の見出しに合わせる。

```markdown
## [YYYY-MM-DD] organize | <短い説明>

- `<wiki>` を推敲・統合・移動しました。
- `wiki/index.md` のリンクを整理しました。
```

最新の履歴が一番上に来るようにする。

### 7. 検証する

- `oxfmt` で編集したMarkdownを整形する。
- `scripts/check_image_links.py` を実行し、Markdown画像リンクが壊れていないことを確認する。
- `rg` で旧パスや旧タイトルへのリンクが残っていないことを確認する。
- `git diff` で `sources/` に意図しない変更がないことを確認する。

## 報告

次を簡潔に報告する。

- 更新、統合、移動、削除したWikiページ
- `wiki/index.md` / `wiki/changelog.md` の更新有無
- 新設・統合・削除したカテゴリがあればその理由
- `sources/` を変更していないこと
- 検証結果
